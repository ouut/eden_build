

ST-GCN Based Action Similarity Comparison System for ARKit Skeleton Data

Version: v2.1
Date: 2026-06-05
Skeleton Data Source: ARKit Body Tracking
Dataset: AIST++ (adapted to ARKit skeleton topology)
Core Algorithm: ST-GCN (Spatial Temporal Graph Convolutional Networks)

---

1. Project Overview

1.1 Objective

This system leverages ARKit Body Tracking on iOS devices to capture real-time 3D human skeleton data, extracts deep spatiotemporal features (Embedding vectors) via an ST-GCN model, and compares them against pre-stored standard action embeddings using similarity metrics. A threshold-based decision determines whether the performed action meets the quality standard.

1.2 Core Pipeline

```
ARKit Skeleton Capture (60fps) → Coordinate Transformation & Normalization → 
Skeleton Retargeting to Model Input → ST-GCN Embedding Extraction → 
Cosine Similarity Comparison → Threshold Decision → Real-time Feedback
```

1.3 Rationale for Technical Choices

Why ST-GCN: The human skeleton naturally forms a graph structure, with physical connections between joints. ST-GCN's graph convolution captures this structured information effectively — the spatial dimension learns inter-joint posture relationships, while the temporal dimension learns dynamic motion trajectories. Compared to CNNs (which flatten skeletons into grids) or RNNs (which ignore spatial structure), ST-GCN provides superior representational capacity for skeleton-based data.

Why Embedding Comparison over Classification: A classifier can only determine "which category" an action belongs to. An Embedding vector, by contrast, retains fine-grained action features and yields a similarity score indicating "how close to the standard," which is essential for action quality assessment.

Why ARKit: ARKit utilizes the built-in camera and LiDAR sensor on iPhones/iPads, requiring no external hardware to obtain 3D skeleton data. It supports 60fps real-time tracking, making it suitable for consumer-facing applications.

2. Core Characteristics of ARKit Skeleton Data

2.1 Joint Structure and Hierarchy

ARKit Body Tracking outputs approximately 19 valid skeleton joints (with slight variations across iOS versions) organized in a tree-shaped hierarchical structure. The root node is located at the hips, from which the spine chain extends upward, the left and right arm branches extend outward, and the left and right leg branches extend downward.

This tree structure naturally fits graph convolutional networks — parent-child joints in the tree form spatial edges in the graph, while the same joint across consecutive frames forms temporal edges. Understanding this structure is essential for defining the adjacency matrix of the ST-GCN.

2.2 Coordinate System Considerations

ARKit employs a right-handed coordinate system: X-axis points right, Y-axis points up, and Z-axis points forward toward the user (camera direction).

Why coordinate transformation is necessary: Most skeleton action recognition datasets (including AIST++) may define coordinate systems differently — some use Z-up, for example. Directly feeding ARKit's raw coordinates into a model trained with a different coordinate convention would cause complete failure. Axis remapping is required to align the coordinate system with the one used during model training.

Furthermore, ARKit's coordinate origin is at the device's initial position, not on the human body. Any minor device movement causes "skeleton position drift." All joint coordinates must be subtracted by the root joint position to transform into a hip-centered local coordinate system. This step eliminates positional variance, allowing the model to focus solely on posture and motion.

2.3 Body Size Variance and Scale Normalization

Different users have different body proportions (height, limb length), causing significant differences in absolute joint coordinates for the same action. A person who is 190cm tall performing a squat will exhibit knee displacement values substantially larger in absolute terms than someone who is 150cm tall.

Principle of bone-length normalization: Compute the average length of all bone segments, then uniformly scale the entire skeleton to a standardized size. After this treatment, the joint coordinate patterns generated by the action "hand touching knee" become consistent regardless of the user's body type. The normalization baseline typically uses the overall average bone length or a specific reference segment such as spine length.

2.4 Data Jitter and Temporal Smoothing

ARKit tracking accuracy is affected by lighting conditions, occlusion, and distance, producing high-frequency micro-jitter in raw data. While this jitter has minimal impact on the overall action features, it introduces noise into the Embedding vector.

Temporal smoothing strategies: Gaussian filtering is suitable for continuous small-amplitude jitter, smoothing trajectories by weighted averaging of neighboring frames. Median filtering is more effective for handling occasional jump anomalies. The smoothing window typically spans 5–9 frames (approximately 80–150 milliseconds), striking a balance between denoising and preserving motion details.

2.5 Occlusion and Confidence

When a user turns around or a limb is occluded by the body, ARKit's position estimation for certain joints becomes unreliable. iOS 14+ provides per-joint tracking confidence scores. A minimum confidence threshold (e.g., 0.5) should be set; frames falling below this threshold should be discarded or filled via interpolation from adjacent frames to prevent feeding erroneous data into the model.

3. Adapting the AIST++ Dataset

3.1 Why AIST++

AIST++ is a high-quality 3D human dance motion database containing diverse movement sequences with precise 3D skeleton annotations. Although it is a dance dataset, the diversity of its skeletal motion patterns enables the trained model to achieve strong generalization capability.

The key challenge is that AIST++ uses SMPL model joint definitions (17 or 25 joints), while ARKit has its own joint definition (19 joints). The number and positional definitions of joints are not identical.

3.2 Skeleton Mapping Strategy

A mapping table from AIST++ joints to ARKit joints must be established. The core principle is semantic alignment — mapping joints with corresponding functions:

· AIST++ Pelvis → ARKit root (hips)
· AIST++ L_Shoulder → ARKit left_shoulder
· AIST++ L_Elbow → ARKit left_arm

For joints present in AIST++ but absent in ARKit (e.g., fine wrist joints), the nearest-neighbor joint position is used as an approximation. For joints present in ARKit but absent in AIST++, estimates are interpolated from existing joints.

This mapping is not a precise one-to-one correspondence, but at the feature level, as long as key joint positions (shoulders, elbows, hips, knees) are aligned, the Embedding extracted by ST-GCN can effectively represent the action.

3.3 Training Data Generation

Once mapping is complete, all AIST++ sequences are converted to the ARKit joint format, followed by unified coordinate normalization and bone-length normalization. This ensures that the feature space learned by the model is that of the ARKit skeleton space. During inference, ARKit raw data can be used directly without further mapping.

4. ST-GCN Model Principles

4.1 Graph Construction

ST-GCN models human skeleton sequences as spatiotemporal graphs:

Spatial dimension: Within a single frame, edges are constructed based on the physical connectivity of the human skeleton. For example, the left shoulder connects to the left upper arm, which connects to the left forearm. These edges form the graph's adjacency matrix, which determines the receptive field of graph convolution — each joint node aggregates information only from its connected neighbor nodes.

Temporal dimension: Connections are established for the same joint across consecutive frames. For instance, the left wrist node in the current frame connects to the left wrist node in the previous and next frames. This enables the network to capture changes in joint positions over time, thereby learning dynamic motion features.

Partitioning strategy: ST-GCN partitions each joint's neighbors into three subsets — the root node itself, centripetal neighbors (nodes closer to the skeleton's center of mass), and centrifugal neighbors (nodes farther from the center of mass). This partitioning assigns different convolution weights, allowing the network to distinguish between "trunk-to-extremity" and "extremity-to-trunk" motion propagation patterns.

4.2 Network Architecture

The network employs a hierarchical stack of ST-GCN Blocks, each containing spatial graph convolution and temporal convolution:

· Shallow layers (64–128 channels): Capture local posture details, such as elbow flexion angle and knee bend amplitude.
· Deep layers (128–256 channels): Capture global motion patterns, such as overall torso rotation and center-of-mass transfer trajectories.

After each Block, feature downsampling is performed (via strided convolution or pooling), progressively compressing temporal and spatial dimensions while expanding the receptive field.

Finally, global average pooling compresses all spatiotemporal information into a fixed-length vector, which is then mapped to the Embedding space through a fully connected layer. The Embedding undergoes L2 normalization, placing it on the unit hypersphere, enabling direct comparison via cosine similarity.

4.3 Why Embedding Dimensions of 128 or 256

This is an empirical choice. Too low a dimension (e.g., 64) loses action details, resulting in insufficient discriminability of similarity scores. Too high a dimension (e.g., 512) increases model parameters, slows inference, and risks overfitting. Dimensions of 128–256 typically represent a balance point between information capacity and efficiency for action similarity tasks.

5. Training Strategy

5.1 Contrastive Learning Approach

The core objective is to make "Embeddings of the same action as close as possible, and Embeddings of different actions as far apart as possible." A pure classification loss can only separate different classes, but intra-class samples may still be widely scattered. Metric learning principles must therefore be introduced.

Triplet Loss: Each training iteration inputs three data samples — an anchor sample, a positive sample from the same class as the anchor, and a negative sample from a different class. The loss function requires that the distance between the anchor and the positive sample be at least a margin value (e.g., 0.3) smaller than the distance between the anchor and the negative sample. This directly optimizes the quality of similarity ranking.

Advantage: It does not depend on a specific threshold but instead ensures that the Embedding space itself possesses good clustering structure. The trained model exhibits a degree of generalization capability even for unseen action types.

5.2 Importance of Data Augmentation

Action recognition tasks are highly susceptible to overfitting to the specific viewpoints or speeds present in the training data. Data augmentation enhances robustness by simulating variations encountered in real-world ARKit usage:

· Spatial augmentation: Random rotation (simulating different user orientations), random scaling (simulating different body types), slight translation (simulating device position offsets).
· Temporal augmentation: Random temporal cropping (simulating speed variations), frame rate perturbation (simulating real-world frame rate fluctuations).
· Noise augmentation: Adding small Gaussian noise to joint coordinates (simulating ARKit tracking jitter).

Augmented data covers a wider range of real-world variations, making the learned features more robust.

6. Inference Pipeline Design

6.1 Sliding Window Mechanism

ARKit continuously outputs skeleton data at approximately 60fps, but the model requires a fixed-length sequence input (e.g., 150 frames, approximately 2.5 seconds). A sliding window buffer is needed to manage the real-time data stream.

Window strategy:

· Maintain a queue with a fixed capacity; new frames are added at the tail, and old frames are evicted from the head.
· At the start of an action, the window gradually fills; during the action, the window slides in real time; at the end of the action, the window gradually empties.
· Set a minimum frame count requirement (e.g., 30 frames, 0.5 seconds). Below this count, inference is not performed, preventing unstable features from overly short action segments.

Padding and truncation: When the actual frame count falls short of the target, duplicate the last frame for padding. When it exceeds the target, perform uniform-interval sampling. This allows the system to handle actions of arbitrary length.

6.2 Similarity Computation and Threshold Decision

After inferring the user action's Embedding vector, compute the cosine similarity against a pre-stored standard action Embedding. Cosine similarity measures the directional agreement of two vectors, ranging from [-1, 1], with 1 indicating perfect identity.

Threshold setting requires calibration on real devices:

1. Collect ARKit data from multiple users correctly performing the standard action.
2. Compute the similarity distribution (mean, standard deviation) of these positive samples.
3. The threshold is typically set as mean − k × standard deviation, with k ranging from 1.5 to 2.0.
4. Simultaneously validate on negative samples: incorrect actions should yield similarity scores significantly below the threshold.

The specific threshold value varies by action type. Simple actions (e.g., standing) can use higher thresholds (0.92–0.95), while complex actions (e.g., dance segments) should adopt more lenient thresholds (0.80–0.88).

6.3 Action Start/End Detection

In a continuous data stream, it is necessary to identify "when an action begins and when it ends"; otherwise, the buffer will mix standing poses with action data. The basic approach is to detect motion magnitude: when the total joint displacement across consecutive frames exceeds a threshold, the action is deemed to have started; when displacement falls back and remains low for a sustained period, the action is deemed to have ended. The stationary frames at the beginning and end serve as natural action boundaries.

7. On-Device Deployment Considerations

7.1 Real-Time Requirements

For fitness experience scenarios, users expect feedback immediately after completing an action. From action completion to receiving the decision result, the ideal latency should be within 300 milliseconds.

Optimization directions include:

· Model quantization: Compressing 32-bit floating-point weights to 8-bit integers can accelerate inference by 2–4×, with typical accuracy loss under 1%.
· Frame-skipping inference: Not every frame requires an Embedding output; inference can be performed every 3–5 frames, with intermediate frames reusing the last result.
· Model architecture simplification: Reducing the number of ST-GCN Blocks or channels trades a slight accuracy decrease for a significant speed gain.

7.2 Format Conversion Pipeline

PyTorch trained model → ONNX intermediate format → Core ML final format. This conversion chain is officially supported by Apple. The key is ensuring that all operators in the model (such as matrix operations in graph convolution) are supported throughout the conversion chain. Complex custom operators may need to be decomposed into basic operators.

8. Special Issues and Countermeasures

8.1 Left-Right Confusion

When a user stands sideways relative to the camera, ARKit may misidentify the left hand as the right hand. This occurs because, from a single viewpoint, left-right symmetric limbs appear visually similar. Mitigation approaches include: leveraging temporal consistency across consecutive frames (joints do not teleport), and requiring the user to face the camera directly during initialization at the start of the action.

8.2 Multi-Action Standard Library Management

In practical applications, comparison against an action library rather than a single standard action is needed. The Embedding vector for each standard action can be stored, and during inference, the user's Embedding is compared against all vectors in the library, with the highest score taken as the match. If the highest score still falls below the threshold, it is classified as an "unknown action" or "unqualified action."

8.3 Model Update Strategy

As user data accumulates, the standard action Embeddings can be continuously optimized: periodically incorporate high-quality, passed user Embeddings into the standard library, using the mean of multiple excellent samples as the new standard. This "centroid drift correction" progressively enhances the representativeness of the standard action, adapting to the postural characteristics of diverse user populations.

9. Evaluation Metrics

Metric Definition Significance in This System
Precision Proportion of truly qualified actions among those judged qualified Low precision means many false passes; users may "trick" the system
Recall Proportion of truly qualified actions that are correctly detected Low recall means qualified users are frequently misjudged, degrading experience
F1-Score Harmonic mean of precision and recall Comprehensive measure of judgment quality
EER (Equal Error Rate) The value at which false rejection rate equals false acceptance rate Measures overall discriminative capability; lower is better
Similarity Mean/Variance Statistical distribution of positive sample similarities Used to establish reasonable threshold intervals
